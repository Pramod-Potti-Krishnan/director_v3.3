## Architecture Overview

### Format Ownership Rules

**Structured Fields** (titles, labels):
- `format_type`: "plain_text"
- `format_owner`: "layout_builder"
- **Flow**: Text Service → plain text → Layout Builder formats

**Styled Fields** (main content, bullets):
- `format_type`: "html"
- `format_owner`: "text_service"
- **Flow**: Text Service → rich HTML → Layout Builder renders as-is

### 90% Threshold Model

Instead of rigid constraints ("5-8 bullets"), flexible limits:
- `max_chars`, `max_words`, `max_lines`
- **Validation**: Hit 90% of ANY limit for visual balance
- **Benefits**: Prevents sparse content AND overcrowding

### Content Flow (v1.1)

```
Director (CONTENT_GENERATION)
    ↓
LayoutSchemaManager.build_content_request()
    • Extracts field_specifications from schema
    • Includes format_type and format_owner for each field
    ↓
Text Service v1.1 (/generate/structured)
    ↓
StructuredTextGenerator.generate()
    • Loops through field_specifications
    • For each field:
        - If format_owner == "text_service":
            - format_type == "plain_text" → _generate_plain_text()
            - format_type == "html" → _generate_html_content()
        - If format_owner == "layout_builder":
            - Always _generate_plain_text() (no formatting)
    • Validates content density (90% threshold for HTML)
    • Returns structured dict matching schema
    ↓
Returns to Director with structured content
    ↓
ContentTransformer.transform_slide()
    • Detects structured content (_is_structured_content() = True)
    • Pass-through WITHOUT parsing or truncation
    • HTML structure preserved
    ↓
Deck Builder / Layout Service
    • Receives structured content
    • For layout_builder fields: applies formatting
    • For text_service fields: renders HTML as-is
    • No more double formatting!
```

## Integration Requirements for Layout Builder

### What Layout Builder Needs to Implement

Layout Builder must respect the `format_owner` specifications:

**For `layout_builder` Fields** (titles, subtitles, labels):
- Receive: Plain text from Text Service
- Responsibility: Apply HTML formatting, fonts, colors, alignment
- Example:
  ```
  Input: "Key Benefits"
  Output: <h1 style="font-family: Arial; color: #333;">Key Benefits</h1>
  ```

**For `text_service` Fields** (main content, bullets):
- Receive: Rich HTML from Text Service
- Responsibility: Render as-is without re-formatting
- Example:
  ```
  Input: <ul><li>Cost savings</li><li>Efficiency</li></ul>
  Output: (renders directly, preserves structure)
  ```

### Format Ownership Detection

```javascript
// Pseudo-code for Layout Builder
function renderField(fieldName, fieldValue, schema) {
    const fieldSpec = schema.content_schema[fieldName];

    if (fieldSpec.format_owner === "layout_builder") {
        // Apply formatting
        return applyFormatting(fieldValue, fieldSpec);
    } else if (fieldSpec.format_owner === "text_service") {
        // Render as-is
        return renderHTML(fieldValue);
    }
}
```

### Executive Summary

After end-to-end testing with Director v3.1 + Text Service v1.1, several layout optimization opportunities were identified. While the Format Ownership Architecture successfully resolved content generation issues, the **layout service needs improvements** in space utilization, text wrapping, and visual balance.

**Testing Date**: 2025-10-21
**Test Scenario**: AI in Healthcare presentation (default scenario)
**Slides Analyzed**: Presentation Agenda (Image #1), Conclusion (Image #2), and others

**Four Main Categories of Improvements**:
1. **Horizontal Space Optimization** - Title/subtitle fields not using full available width
2. **Vertical Space Optimization** - Content boxes not expanding to fill available height
3. **Inter-Section Spacing Standards** - Inconsistent white space between sections
4. **Text Wrapping** - Improper wrapping causing truncation despite available space

### Category A: Horizontal Space Optimization

**Issue Observed** (Image #1 - Presentation Agenda):
- Slide title and subtitle are significantly shorter than available horizontal space
- Content appears left-aligned with large empty space on right
- Not utilizing the 32x16 grid system effectively

**Root Cause**:
- Title/subtitle fields may have fixed width constraints
- Not extending to penultimate grid column as they should

**Recommendation**:
```
CURRENT: Title extends to ~grid column 12 (out of 32)
DESIRED: Title should extend to grid column 31 (penultimate column)

Layout Rule: Title/subtitle fields should span:
- Start: Grid column 2 (1 grid left margin)
- End: Grid column 31 (1 grid right margin)
- Total width: 30 grid columns (93.75% of slide width)
```

**Implementation Guide**:
```css
/* Pseudo-CSS for layout service */
.slide-title, .slide-subtitle {
    grid-column: 2 / 31;  /* Spans columns 2-31 */
    width: 100%;           /* Fill allocated grid space */
    text-align: left;
}
```

**Visual Impact**:
- ✅ Better space utilization (93.75% vs ~37.5%)
- ✅ Professional appearance
- ✅ Prevents awkward empty space on right side

### Category B: Vertical Space Optimization

**Issue Observed** (Image #1 - Presentation Agenda):
- Main content text box only uses ~40% of available vertical space
- Could extend downward by at least 4 grid rows
- Slide appears sparse and unbalanced

**Root Cause**:
- Text box height may be fixed or insufficiently calculated
- Not adapting to actual content length vs available space
- Missing vertical fill logic

**Recommendation**:

   - Start: Grid row 5 (after header + top margin)

**Implementation Guide**:

/* Pseudo-CSS for layout service */

**Special Case - Bullet-Only Content**:

### Category C: Inter-Section Spacing Standards

**Issue Observed** (Image #1 - Presentation Agenda):
- Inconsistent spacing between slide sections
- Some gaps appear too large (3-4 grids)
- No clear spacing standard being followed

**Root Cause**:
- Layout service may use fixed pixel spacing instead of grid-based spacing
- Different layouts have different spacing values
- No unified spacing design system

**Recommendation - Grid-Based Spacing System**:

```
STANDARD SPACING RULES (based on 32x16 grid):

1. Between Major Sections (Title ↔ Content, Content ↔ Footer):
   - Standard: 1 grid row vertical spacing
   - Maximum: 2 grid rows (exceptional cases only)

2. Within Content Sections (Paragraph ↔ Bullet List):
   - Standard: 0.5 grid rows (half-grid spacing)
   - Maximum: 1 grid row

3. Horizontal Margins:
   - Left margin: 1 grid column (column 2 start)
   - Right margin: 1 grid column (column 31 end)
   - Between columns in multi-column layouts: 1 grid column

4. Exception Cases for 2-Grid Spacing:
   - Visual emphasis needed (e.g., before key insights box)
   - Section break that requires clear separation
   - Designer's discretion for specific layouts
```

**Example Spacing Breakdown**:
```
Grid Row 1:    [Top Margin - 1 grid]
Grid Rows 2-3: [Title + Subtitle - 2 grids]
Grid Row 4:    [Spacing - 1 grid] ← STANDARD
Grid Rows 5-13: [Main Content - 9 grids]
Grid Row 14:   [Spacing - 1 grid] ← STANDARD
Grid Row 15:   [Footer - 1 grid]
Grid Row 16:   [Bottom Margin - 1 grid]
```

**Visual Impact**:
- ✅ Consistent professional appearance across all layouts
- ✅ Predictable visual rhythm
- ✅ Better space utilization (less wasted white space)

### Category D: Text Wrapping & Truncation

**Issue Observed** (Image #2 - Conclusion: Key Takeaways):
- Text is not wrapping within container boundaries
- Shows truncation despite adequate horizontal space available
- Subtitle text appears cut off prematurely

**Root Cause**:
- Text wrapping logic may be disabled or misconfigured
- Container width constraints not properly communicated to text renderer
- Possible CSS `white-space: nowrap` or `overflow: hidden` without wrapping

**Recommendation**:

```
TEXT WRAPPING RULES:

1. For All Text Fields (Title, Subtitle, Content):
   - Enable automatic wrapping: CSS `word-wrap: break-word`
   - Respect container boundaries: CSS `overflow-wrap: break-word`
   - Multi-line support: CSS `white-space: normal`

2. Container Width Communication:
   - Layout service must provide exact container width to text renderer
   - Text service generates content within these bounds
   - Layout service respects generated content length

3. Truncation Rules (ONLY when necessary):
   - Title fields: Truncate ONLY if exceeds 3 lines
   - Subtitle fields: Truncate ONLY if exceeds 2 lines
   - Content fields: NEVER truncate (use scrolling if needed)
   - NO ellipsis ("...") - already fixed in v3.1

4. Line Height Considerations:
   - Title: 1.2-1.3x line height for readability
   - Body content: 1.5x line height for readability
   - Ensure line height doesn't cause overflow
```

**Implementation Guide**:
```css
/* Pseudo-CSS for layout service */
.slide-title, .slide-subtitle, .content-text {
    word-wrap: break-word;        /* Wrap long words */
    overflow-wrap: break-word;    /* Break words if necessary */
    white-space: normal;           /* Allow multi-line */
    hyphens: auto;                /* Optional: hyphenation */
}

.slide-title {
    max-height: calc(1.3em * 3); /* Max 3 lines */
    overflow: hidden;             /* Hide if exceeds */
}

.slide-subtitle {
    max-height: calc(1.3em * 2); /* Max 2 lines */
    overflow: hidden;
}

.content-text {
    max-height: 100%;            /* Fill container */
    overflow-y: auto;            /* Scroll if needed */
}
```

**Visual Impact**:
- ✅ No truncated text when space is available
- ✅ Professional multi-line wrapping
- ✅ Improved readability

### Format Ownership Clarification (Reinforced)

To avoid confusion during layout service improvements, **reiterating the format ownership rules**:

**Layout Service Owns HTML Formatting For**:
- Slide titles
- Slide subtitles
- Footer elements
- Labels and headers
- Contact information
- Any field marked `format_owner: "layout_builder"`

**Layout Service Responsibilities**:
- Apply fonts, colors, sizes, alignment
- Add HTML tags (`<h1>`, `<p>`, `<span>`, etc.)
- Control visual styling with CSS classes
- Receives **plain text** from Text Service

**Text Service Owns HTML Formatting For**:
- Main content sections (>20 words)
- Bullet lists (`<ul>`, `<ol>`)
- Text boxes with rich content
- Insight boxes, quote boxes
- Any field marked `format_owner: "text_service"`

**Text Service Responsibilities**:
- Generate rich HTML with structure
- Include `<ul><li>` for bullets
- Include `<p>` for paragraphs
- NO visual styling (no CSS, no font specs)
- Layout service renders this HTML **as-is**

**Why This Matters**:
- ✅ Prevents double formatting (layout service adding bullets to already-bulleted HTML)
- ✅ Prevents empty bullets (text service controls HTML structure)
- ✅ Prevents format conflicts (clear separation of concerns)

### Grid-Based Design System Reference

For layout service developers, here's the complete grid system specification:

**Base Grid**: 32 columns × 16 rows

**Horizontal Grid (32 columns)**:
```
Column 1:     Left margin (3.125% of slide width)
Columns 2-31: Content area (93.75% of slide width)
Column 32:    Right margin (3.125% of slide width)

Grid column width: 1/32 = 3.125% of slide width
```

**Vertical Grid (16 rows)**:
```
Row 1:        Top margin (6.25% of slide height)
Rows 2-15:    Content area (87.5% of slide height)
Row 16:       Bottom margin (6.25% of slide height)

Grid row height: 1/16 = 6.25% of slide height
```

**Standard Layout Zones**:
```
HEADER ZONE:
- Grid rows: 2-3 (title + subtitle)
- Height: 2 rows = 12.5% of slide height

CONTENT ZONE:
- Grid rows: 5-14 (main content after 1-grid spacing)
- Height: 10 rows = 62.5% of slide height

FOOTER ZONE:
- Grid row: 15 (footer content)
- Height: 1 row = 6.25% of slide height
```

**Spacing Values**:
- 1 grid row vertical = 6.25% slide height (~45-50px on 1080p)
- 2 grid rows vertical = 12.5% slide height (~90-100px on 1080p)
- 1 grid column horizontal = 3.125% slide width (~60px on 1920p)

**Why Grid-Based Design?**
- ✅ Consistent spacing across all layouts
- ✅ Responsive to different slide sizes
- ✅ Easier to calculate available space
- ✅ Professional, predictable visual rhythm
- ✅ Aligns with modern web design practices (CSS Grid)
